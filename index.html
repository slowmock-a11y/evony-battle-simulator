About

“Evony — A war game full of adults acting like children throwing away tons of money to fill in that void of acceptance in their lives.”
— Derrick Defies

The Evony Battle Simulator models march-vs-march PvP combat using community-researched battle mechanics. Configure two armies, set buff percentages, and watch the engagement unfold step-by-step on a spatial battlefield.

Work in progress: The battle mechanics are based on the best community research available, but many facts are still unclear, missing, or approximated. If you have corrections, better data, or suggestions, please open an issue, join the discussion, or submit a pull request — all contributions are welcome.

Key Features

  • Army Configuration — Set troop counts per type and tier (T1–T16) for attacker and defender, with ATK/DEF/HP buff percentages per troop type.
  • Step / Round / Full Playback — Advance the battle one attack at a time, one full round, or run to completion, with adjustable speed.
  • Spatial Battlefield — A 1500-unit linear field with troop movement, range indicators, and attack arrows. (Field length is currently under investigation — a competing hypothesis says field = max base range + 200.)
  • Battle Log & Summary — Filterable log of every attack event, plus a side-by-side loss comparison panel.

Not Yet Modeled

These real-game mechanics are excluded from the current simulator:

  • Generals (base stats, ascending stars)
  • General Skills & Specialties
  • Equipment on generals
  • Research / Technology buffs
  • Civilization & Alliance buffs
  • Monarch Gear & Sub-city buffs
  • Wounded / Killed ratio
  • Wall / Trap mechanics (city attacks only)
  • Rally mechanics (multi-march)

Battle Mechanics Fact Sheet

Reference for the simulation rules. Sources: community research, in-game observation, and community sites listed below.

Troop Types

Four troop types, each with tiers T1 through T16 (T16 unlocks at Workshop L45). Each type fills a distinct battlefield role:

TypeRoleKey TraitSpeedRange
GroundTankHighest DEF (~3× ATK), highest HP (~6× ATK)35050
RangedBalancedATK ≈ DEF, moderate HP (~2× ATK)100500
MountedStrikerHighest ATK, fastest unit in the game30050
SiegeGlass CannonLowest stats overall, extreme range75 (T1–T15), 76 (T16)900–1400

Speed and range are constant across all tiers for a given type. Only ATK, DEF, and HP scale with tier. Siege has two exceptions — its range increases in tier brackets (T1–T4: 900, T5–T8: 1000, T9–T10: 1100, T11–T12: 1200, T13–T16: 1400), and its speed steps from 75 to 76 at T16. Values use actual game-DB data (UI displays inflated numbers).

Notable: Siege ATK = Siege HP at every tier T1–T15. Exception at T16: ATK 4,440 ≠ HP 4,400. Ground HP is roughly 6× its ATK at every tier.

Sources: Theria Games — Troop Base Stats • Evony Guide Wiki — Troop Types

Phase Order (Attack Sequence)

Each round is divided into four phases. Troops attack sequentially by actual speed (fastest first). When speeds tie, the defender acts first:

PhaseTypeSpeedRationale
1Ground350Fastest actual speed
2Mounted300Second fastest
3Ranged100Third
4Siege75Slowest

Within each phase: The troop taking its turn strikes first; the target's surviving troops counter-strike after. Counter-strikes are range-gated — a troop can only counter-strike if the attacker is within its own actual range (e.g. Ground and Mounted at range 50 cannot counter-strike Ranged or Siege firing from farther away).

Within a side (same type, multiple tiers): Higher tiers attack before lower tiers, dealing damage with better stats first.

Sources: EvonyTips — How Battles Work • Server 806 — Battle Mechanics

Battlefield & Movement

Linear battlefield. Attacker starts at position 0, defender at the far end. Each troop type has its own independent position (8 positions total: 4 per side).

Field length is disputed. The simulator currently assumes a fixed length of 1,500 units, but this value has no primary game-database citation. A competing community formula proposes a dynamic field: max(base range of any troop type present) + 200, which would range from 250 (pure melee) to 1,600 (with T13+ Siege present). The two hypotheses are mutually exclusive — no troop composition yields exactly 1,500 under the dynamic formula. Until an in-game test resolves this, the 1,500 value should be treated as provisional. See the battlefield-length investigation page for the two hypotheses, the differential test protocol, and how to share your findings.

Attacker (0) ————— 1,500 units* ————— Defender (1500)
*provisional — see disputed-field note above

Each round, before attacking, each troop type evaluates movement:

  • Check: Is any enemy unit within range from current position?
  • Yes → Hold. Stay at current position and fire at the highest-priority target in range.
  • No → Advance. Move forward by speed units toward the enemy. Fire if a target is now in range.

Key rules:

  • Speed = distance per round. Ground moves 350 units/round, Mounted 300, Ranged 100, Siege 75.
  • Direction is always forward. Troops never reverse or retreat.
  • Minimum-distance movement: Troops move only the minimum distance needed to bring the highest-priority target into actual range. They don't blindly advance full speed.
  • Effective range = speed + range: Determines detection distance for movement decisions, not firing distance.
  • Hold triggers on any enemy in range, not just the priority target.
  • Melee collision: When opposing troops would pass through each other, they stop at melee range (50 units apart).
  • Engagement lock: Once attacking a target type, troops stay locked on until that type is eliminated or a higher-priority type enters effective range.

Typical engagement timeline (assuming the provisional 1,500-unit field, both sides identical):

  • Round 1: Siege detects approaching troops. Ground/Mounted advance.
  • Round 2: Mounted pairs close to melee near midfield.
  • Round 3: Ground pairs lock in melee.
  • Round ~5: Ranged reach engagement range.

Timeline reflects the 1,500-unit assumption. Under the competing dynamic-field hypothesis, matchups without T13+ Siege would collapse the cascade substantially — e.g. Ranged pairs could engage as early as Round 1 on a 700-unit field. All other values use actual game-DB data (UI displays inflated numbers for some stats).

Sources: EvonyTips — How Battles Work • Theria Games — Layers Guide

Damage Formula

damage = troopCount × ATK × modifier × ATK / (ATK + DEF)
kills  = floor(damage / target_HP)
  • troopCount — surviving count of the attacking layer
  • ATK — per-troop attack stat of the attacker (base × (1 + buff%))
  • DEF — per-troop defense stat of the target (base × (1 + buff%))
  • modifier — type matchup multiplier (see below)
  • target_HP — per-troop HP of the target (base × (1 + buff%))

The ATK/(ATK+DEF) ratio ensures damage is always > 0 (no zero-damage edge cases), and high DEF reduces damage proportionally rather than by flat subtraction. The ratio is always between 0 and 1.

Buff percentages are additive: a 50% ATK buff on a T14 Mounted (base 6670 ATK) yields 6670 × 1.50 = 10,005 effective ATK.

Sources: Evony Guide Wiki — Battle Mechanics • Evony Guide Wiki — Buff/Debuff Guide

Battle End Condition

The battle ends when one side has zero troops remaining. The surviving side is declared the winner.

Whether field march-vs-march battles have a fixed round cap (community often cites 4 rounds for certain battle types) is unconfirmed for open-field PvP. This simulator runs until one side is eliminated.

Battlefield Length — Open Investigation

This page documents an unresolved factual uncertainty in the simulator: the length of the battlefield. The simulator currently uses a fixed value of 1,500 units, but that value is disputed. Until an in-game test resolves the question, simulated outcomes for matchups where engagement timing matters may be systematically off.

Status (2026-04-17): open question, confidence LOW. Test protocol defined, awaiting in-game runs.

Why this matters

Battlefield length determines how many rounds of movement each troop type needs before it can fire. A longer field means slower or shorter-range types spend more rounds being shot at before they close to melee. That changes survivor counts in every asymmetric matchup.

The simulator hard-codes BATTLEFIELD_LENGTH = 1500 in both js/battle-engine.js and js/battlefield.js. The mechanics fact sheet previously labeled 1500 as “confirmed,” but that label traces to a single community source chain (evonyanswers.com citing @DerrickDefies) without a primary game-database reference.

The two hypotheses

H1 — Fixed field:

BATTLEFIELD_LENGTH = 1500  (constant, independent of composition)

H2 — Dynamic field on base range:

BATTLEFIELD_LENGTH = max(base_range(layer) for layer in all_layers_present) + 200

The H2 formula uses base range values (Ground 50, Mounted 50, Ranged 500, Siege 900–1400 by tier band). Range-modifying buffs (research, skills, gear) are explicitly excluded from the formula.

Under H2, field length is matchup-dependent:

Highest base range presentField under H2
Melee only (50)250
+ Ranged (500)700
+ Siege T1–T4 (900)1,100
+ Siege T5–T8 (1,000)1,200
+ Siege T9–T10 (1,100)1,300
+ Siege T11–T12 (1,200)1,400
+ Siege T13–T16 (1,400)1,600

Notable: no troop-tier combination yields exactly 1,500 under H2. This means at least one of the two hypotheses must be wrong — they cannot both be correct.

Why a differential test instead of an absolute prediction

In-game battle reports on this server expose only start counts and final survivor counts per layer. There is no round counter, no timeline, no per-round damage log. An absolute-prediction test (“the sim says 743 Ranged survive; did reality match?”) depends on matching every buff, every coefficient, every tier stat exactly. Too brittle.

Instead, we use a differential test: run two in-game battles that differ by one ingredient, where H1 and H2 disagree about whether that ingredient matters. Buffs held constant between the two runs cancel out. Only the hypothesis-relevant variable drives the delta.

Test protocol — the siege-ingredient test

Run A (baseline):

  • Attacker: 1000 Ranged T1
  • Defender: 1000 Ranged T1
  • Everything else: zero (no Ground, no Mounted, no Siege, no walls, no traps)

Run B (ingredient added):

  • Attacker: 1000 Ranged T1 + 1 Siege T13 (single unit, purely to stretch the field under H2)
  • Defender: 1000 Ranged T1 (unchanged)
  • Everything else: zero

The Siege T13 unit is chosen because:

  • Its base range (1,400) is the maximum in-game, so it maximises H2's field-length jump: 700 → 1,600.
  • A count of 1 contributes negligibly to damage output (1 × 2,780 ATK << 1000 × 130 × 0.29 Ranged mutual output), so its direct combat effect is below the noise floor.
  • T13 is more accessible than T14–T16 for most players; T14/T15/T16 would also work and are interchangeable from a range standpoint (all are 1,400).

Predicted outcomes

H1: field = 1500H2: field = max_base_range + 200
Run A (Ranged only) Field 1500. Ranged fires starting round ~5, battle continues to exhaustion. Field = 500+200 = 700. Ranged fires starting round 1. More firing rounds → much deeper mutual attrition.
Run B (Ranged + 1 Siege T13) Field 1500 (unchanged). Single siege unit is irrelevant. Essentially identical to Run A. Field = 1400+200 = 1,600. Ranged now needs ~6 rounds to close. Fewer firing rounds → less attrition than Run A.
Δ (B − A) Ranged casualties ≈ 0 (within RNG noise) Clearly negative (Run B preserves more Ranged than Run A)

The adjudicating signal: does adding a single Siege T13 to one side of a pure-Ranged matchup change Ranged survivor counts? H1 says no. H2 says yes, materially.

Controlling for buffs

Both A and B runs must use:

  • The same general on both sides if possible (alliance partner uses the same level/skill configuration for both), OR the same two generals across runs (do not swap between A and B).
  • No Range-affecting research, skills, gear, or sub-city tech active on either side. H2 is defined on base range; if Range is buffed, the field length under H2 is no longer cleanly predictable from the formula.
  • Preferably: low-tier general, no specialties activated, no monarch gear, no active spiritual beast / dragon bonuses. This minimises absolute-magnitude variance but is not strictly required as long as buffs are identical between runs.

ATK/DEF/HP buffs do not need to be zeroed. They scale both sides' firing output identically and therefore scale total casualties uniformly. They do not interact with field length. As long as they are held constant between Run A and Run B, the differential signal is preserved.

Noise reduction

A single battle pair (A, B) can be dominated by RNG, accidental asymmetry, or misconfiguration. Run each scenario three times (three A battles, three B battles). Average the Ranged casualty count.

  • H1 prediction: |mean_B − mean_A| / mean_A < 2% (the single siege is negligible).
  • H2 prediction: Δ on attacker and defender Ranged are both positive (more Ranged survive when siege is present) and at least one is ≥ 10% of mean_A.

Running the test — checklist

1. Pre-test preparation

  • Identify every source of Range buff available on the server (research, general skills, specialties, gear, sub-city buffs, alliance tech, spiritual beast, dragon, civilisation bonus, monarch equipment) and confirm each can be disabled or is not currently active on the test account.
  • Recruit an alliance sparring partner willing to run six battles with identical setups. Confirm their general/buff configuration is available and will be held constant across runs.
  • Screenshot the general, skill activation state, and any visible buff panel on both accounts before the test begins. Store screenshots alongside the results.

2. Run A — baseline, three repetitions

  • A1: 1000 Ranged T1 attacker vs 1000 Ranged T1 defender. Record attacker/defender Ranged survivors.
  • A2: same setup as A1. Record.
  • A3: same setup as A1. Record.

3. Run B — siege ingredient added, three repetitions

  • B1: (1000 Ranged T1 + 1 Siege T13) attacker vs 1000 Ranged T1 defender. Record.
  • B2: same setup as B1. Record.
  • B3: same setup as B1. Record.

4. Analysis

  • Compute mean Ranged survivors across A1–A3 and across B1–B3 on each side.
  • Compute Δ = mean_B − mean_A for attacker Ranged and for defender Ranged.
  • Apply the H1 vs H2 thresholds from “Noise reduction” above.

Known risks / caveats

  • Alliance-partner cooperation needed. Without a willing sparring partner, the player must find a target whose defence composition they know exactly. The differential design tolerates some unknowns as long as they are constant between A and B.
  • Hidden round cap. If march-vs-march has an undocumented round cap, H1's Run A might show very few casualties (no Ranged fires before cap expires). That is itself a useful data point and strengthens the test rather than invalidating it.
  • Ranged-buff leakage. If a buff that affects Range is overlooked, the H2 formula input is wrong. The pre-test preparation step asks you to enumerate and disable them.
  • Damage-formula RNG on this server. If there is RNG, the three-run average mitigates but does not eliminate it. If results are borderline, extend to 5 or 7 runs.
  • MER (buffed range) variant not addressed. If the field uses buffed MER rather than base range, the formula is again different. If Run A and Run B behave per H2 but absolute magnitudes disagree with predicted values, MER may be the next hypothesis to test.
  • Static vs dynamic layer death. The current test cannot distinguish whether field length depends on “layers present at start” or “layers alive now.” A multi-phase follow-up test is needed for that.
